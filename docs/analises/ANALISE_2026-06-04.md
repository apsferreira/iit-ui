# Análise — iit-ui

**Data:** 2026-06-04 | **Categoria:** infra | **Método:** auditoria holística single-agent | **Baseline:** sem baseline anterior.

---

## Resumo executivo

`@iit/ui` é uma biblioteca de componentes React (web + React Native) bem estruturada para seu estágio — 9 componentes estáveis, tokens tipados, Storybook, CI com GHCR e publicação no GitHub Packages. A maturidade geral está em nível **alpha funcional**: arquitetura limpa, TypeScript strict sem `any`, bom padrão de acessibilidade (aria-*, focus trap, WCAG 2.1 AA declarado). Os riscos mais graves são três falsos negativos nos testes (classes que os testes asseguram não existem no código real — `text-red-400`, `text-emerald-400`, `iit-fadeIn`), ausência total de cobertura de testes medida via `@vitest/coverage-v8` (threshold não configurado), e o CI que referencia `k8s/deployment.yaml` que foi deletado do repo (step falho silencioso em cada push para main). Secundariamente, o Card tem cores dark-mode hardcoded (`#111118`, `#2A2A38`) que quebram a consistência do token system — o único componente que não usa `surface.*`/`text.*`. Sem baseline anterior para comparação.

## Arquitetura

**Forças:** estrutura de módulos clara (`src/components/<Name>/Name.tsx + index.ts`); separação web/native por sufixo `.native.tsx`; token system tipado (`as const`) com `Colors` e `Typography` exportados; `cn()` correto com `twMerge + clsx`; exports condicional `react-native` no `package.json` bem configurado; Tailwind preset exportável.

**Gaps:**
- `tailwind.config.ts` exportado na forma fonte TS — apps sem suporte a TS em config (CRA, Next.js < 13 com config JS) falharão ao importar `@iit/ui/tailwind` (P2).
- `Card.tsx` rompe o token system — único componente com paleta própria dark (`#111118`, `#2A2A38`, `#F4F4F8`, `#A1A1B5`) não derivada de `surfaceDark.*` (P1).
- `class-variance-authority`, `clsx`, `tailwind-merge` bundleados — deveriam ser `external` no Rollup (P2).

## Backend & Segurança

N/A: biblioteca frontend sem backend, sem chamadas de API, sem autenticação, sem secrets hardcoded no src/. `.npmrc` contém apenas o registry URL sem token — correto.

## Testes

**Forças:** todos os 9 componentes têm arquivo `.test.tsx`; uso de `@testing-library/react` e `@testing-library/user-event`; `aria-*` e `role` verificados; `EmptyState.test.tsx` mais completo (6 casos, `userEvent` assíncrono).

**Gaps críticos:**
- `Button.test.tsx:33` — `toContain('text-red-400')` falso: classe real é `text-red-600` (P0).
- `Badge.test.tsx:13` — `toContain('text-emerald-400')` falso: classe real é `text-[#00A87E]` (P0).
- Nenhuma configuração de `coverage.thresholds` no `vitest.config.ts` — gate 80% inexistente (P1).
- Zero testes para implementações `.native.tsx` (P2).
- `vitest.config.ts` sem `exclude` explícito para `*.stories.tsx` e `*.native.tsx` (P2).
- Nenhum teste de integração Esc/Tab no Modal — `Modal.test.tsx` não cobre focus trap (P2).

## DevOps

**Forças:** CI com GHCR + multi-tag (branch, sha, latest); Dependabot configurado para npm e github-actions; `publish-npm.yml` com trigger por tag semântica e `workflow_dispatch`.

**Gaps:**
- `ci.yml:66` referencia `k8s/deployment.yaml` deletado — step falha em toda push para main (P0 — CI quebrado silenciosamente).
- CI não executa `npm test` nem `lint` — falsos negativos passam (P1).
- Actions sem SHA pin — supply chain risk (P1).
- `publish-npm.yml` publica sem rodar testes (P1).
- `sed -i` em macOS vs Linux: `sed -i` sem extensão funciona em Linux mas falha em macOS — irrelevante para `ubuntu-latest` runner, mas vale nota.
